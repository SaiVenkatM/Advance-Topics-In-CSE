    I'll analyze this academic paper on Bladerunner, a system used at Facebook to deliver relevant social graph updates to user devices efficiently and in real-time.

    Key aspects of Bladerunner's design and architecture:

    1. Core Components:
    • BRASS (Bladerunner Application Stream Servers): Backend streaming servers that process updates on a per-client and per-application basis
    • Pylon: A simple, best-effort pub/sub system for disseminating updates to BRASSes
    • BURST: A request-stream protocol for device-BRASS communication
    1. Unique Design Choices:
    • Best-effort delivery with reliable failure detection, rather than guaranteed delivery
    • Separate BRASS instances per application for simpler implementation
    • Propagates only update events (not data) via Pylon to reduce cross-region bandwidth
    • BURST protocol supports long-running streams across multiple network hops
    1. Key Benefits:
    • Reduced backend server load by eliminating unnecessary polling
    • Lower device processing overhead and last-mile bandwidth usage
    • Faster update delivery compared to polling-based approaches
    • Better scalability for applications with high update rates
    1. Scale Metrics (from production):
    • Can handle >1B update deliveries per minute at peak
    • ~75% of request-streams receive no updates
    • Average latencies: ~240ms for basic updates, ~2s for LiveVideoComments
    • Handles millions of dropped connections per minute gracefully
    1. Real-World Impact:
    • When LiveVideoComments switched from polling to Bladerunner:
      • CPU load reduced by 0.4% on average (saving thousands of hosts)
      • TAO queries reduced by 0.7%
      • Update delivery latency decreased by ~30%
      • Peak load reductions: 5% on TAO IOPS, 2.5% on web servers
    1. Failure Handling Philosophy:
    • Prioritizes fast best-effort delivery over guaranteed delivery
    • Focuses on reliable failure detection and notification
    • Pushes reliability implementation to applications when needed
    • Leverages existing persistent storage (TAO) for recovery

    The paper effectively demonstrates how Bladerunner solves the challenging problem of delivering relevant real-time updates at massive scale while minimizing infrastructure load and bandwidth usage.
